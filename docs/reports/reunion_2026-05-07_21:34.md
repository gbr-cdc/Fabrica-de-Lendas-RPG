# Reunion Report: 2026-05-07

## Problems Identified
1. **Destructive Updates**: It was observed that updating a parent section using `ref_manager.py --update` replaced the entirety of the section until the next equal or higher level header. This inherently deleted all subsections and their associated tags.
2. **Single Operation Limitation**: The `ref_manager` tool was designed to halt after processing the first write operation (`--update`, `--create`, or `--delete`). Agents were not aware of this and were attempting to string multiple operations in a single command, leading to ignored commands.

## Changes Made
Updated `[AGENT.REF_MANAGER]` rules in `agent_rules.md` to prevent these issues in the future:
- **Single Operation Constraint**: Documented that `ref_manager` only supports one write operation per command, requiring separate calls for multiple changes.
- **Avoid Destructive Updates**: Added warnings that updating a parent section destroys subsections. Instructed agents to always prefer using `--create` with a specific sub-tag when adding new information.
- **Granular Updates**: Specified that if `--update` is necessary, agents must target the most specific tag possible to minimize collateral damage.
